# Highlight Studio v1.0 Final Structure Baseline

## 1. Project Overview

Highlight Studio is a local-first highlight video production tool for photo-based event videos.

Purpose:

- Import event photos.
- Organize photos into projects and timelines.
- Apply templates, captions, transitions, and output settings.
- Render MP4 files locally through FFmpeg.
- Provide both browser and Electron desktop execution.

Execution modes:

- Browser: `npm start`, then open `http://localhost:4000`.
- Electron: `npm run electron`, which opens a desktop BrowserWindow and starts or reuses the local server.
- Installer build: prepared through the Electron build structure already present in the project.

Local processing principles:

- Media processing runs on the local PC.
- Render jobs, upload temp files, and outputs are local filesystem state.
- No Firestore/Firebase dependency is used for Highlight Studio.
- No OpenAI or cloud AI dependency is used for local AI rules.
- SportsLink integration remains external and only launches/links to Highlight Studio.

External dependency policy:

- FFmpeg is used for local rendering.
- Electron is used for desktop packaging/runtime.
- Express serves the local app and API.
- External network services are not required for core render/project workflows.

## 5. Render Architecture

Render flow:

```text
POST /api/render
-> upload middleware
-> parse project payload
-> create render job
-> render job store
-> enqueue render job
-> queue controller
-> render executor
-> FFmpeg engine
-> output file
-> public job status
```

Core responsibilities:

- Queue: `render-queue-operations.js` and `render-queue-controller.js` control enqueue/remove and active job execution.
- Job store: `render-job-store.js` owns the single job Map, queue array, and active job id.
- Job utilities: `render-job-utils.js` owns status/log updates and terminal cleanup scheduling.
- Executor: `render-executor.js` translates project/timeline/photos into render work and output metadata.
- FFmpeg: `ffmpeg-engine.js` owns process spawning, progress/error handling, encoder detection, and encoder argument selection.
- Cancellation: `POST /api/render/cancel/:jobId` marks queued/active jobs canceled; active FFmpeg process is terminated with `SIGTERM`.
- Progress: public job shape exposes status, progress, current photo, encoder, errors, and output metadata.
- Output: completed render writes MP4 to the configured output directory and exposes it through `/outputs` and `/api/outputs`.

GPU behavior:

- Auto encoder detection checks available encoders.
- CPU fallback remains the safe baseline.
- FFmpeg GPU encoding and Electron Chromium GPU settings are separate concerns.

## 6. Electron Structure

Main files:

- `desktop/main.js`: Electron app lifecycle, BrowserWindow creation, server startup/reuse, protocol handling, settings UI, shutdown wiring.
- `desktop/preload.js`: preload bridge for desktop-safe APIs.

Runtime behavior:

- Electron starts or reuses the local Express server at `http://127.0.0.1:4000`.
- The desktop window loads the local app rather than exposing a browser address bar.
- `highlightstudio://open` protocol registration is prepared for launcher integration.
- `before-quit` routes shutdown through `shutdownServer`.

Shutdown flow:

```text
Electron before-quit
-> serverModule.shutdownServer(...)
-> cancelAllRenderJobs(reason)
-> HTTP server close
-> app.quit()
```

If Electron reuses an already-running server, it does not own or close that server.

## 8. Final Structure Changes

Original structure had most behavior concentrated in `server.js`.

Major reductions and separations:

- `server.js`: reduced from roughly 1400+ lines during early stabilization to 89 lines at this baseline.
- `server/bootstrap.js`: owns application assembly and route/service registration.
- Static template data moved to `server/default-templates.js`.
- Local AI rules moved to `server/ai-local-rules.js`.
- License helpers moved to `server/license-helpers.js`.
- Output helpers moved to `server/output-helpers.js`.
- Template helpers moved to `server/template-helpers.js`.
- Upload filename helpers and multer middleware moved to dedicated modules.
- Project helpers and project read/write/load routes moved to dedicated modules.
- System, AI, template, output, render, project, and legacy routes moved to route modules.
- Render job store, job utilities, queue operations, queue controller, executor, and FFmpeg engine were split into explicit modules.
- Legacy renderer and legacy route were split without deleting compatibility.
- Shutdown manager was added as a small shared lifecycle utility.

The result is a thin entrypoint plus modular services/routes while preserving API compatibility and local behavior.

## 10. Release Limitations

The following are verification-environment limitations, not known code defects:

- Browser file picker/upload was not fully automated through visible UI controls because the available browser control surface did not expose a safe file chooser workflow.
- Node `Ctrl+C` / SIGINT direct delivery was blocked by the execution backend used for automated verification.
- Electron UI close was verified through a window close message and logs, but one standalone Electron run showed default storage paths in startup logs because desktop settings can override environment-derived storage paths. No state-changing Electron workflow was run against default user data.

These should be confirmed once on an actual user machine before declaring Release Ready.

## 11. Release Decision

Current status:

**Release Candidate v1.0**

Promotion to **Release Ready** requires:

1. Confirm Browser upload through the real file picker in an actual user/browser environment.
2. Confirm `Ctrl+C` shutdown once in a normal terminal.
3. Confirm Electron UI close once in the installed/desktop runtime, including `before-quit` and port release.

No additional code fix is required by the current documented baseline.
